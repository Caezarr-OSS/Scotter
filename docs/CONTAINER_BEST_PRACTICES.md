# Container Best Practices with Scotter

This guide provides best practices for working with container configurations generated by Scotter.

## General Container Best Practices

### 1. Keep Images Small

Smaller images have numerous benefits:
- Faster build and deployment times
- Reduced attack surface
- Lower resource usage

Tips for keeping images small:
- Use Alpine-based images when possible
- Remove unnecessary packages and files
- Use multi-stage builds
- Avoid installing development tools in the final image

### 2. Use Specific Base Image Tags

Always use specific version tags for base images instead of `latest`:

```dockerfile
# Good
FROM golang:1.21-alpine

# Avoid
FROM golang:latest
```

This ensures reproducible builds and prevents unexpected changes.

### 3. Optimize Layer Caching

Structure your Dockerfile to maximize caching benefits:

```dockerfile
# Copy dependency files first
COPY go.mod go.sum ./
RUN go mod download

# Then copy source code (changes more frequently)
COPY . .
```

### 4. Run as Non-Root User

For security, run containers as a non-root user:

```dockerfile
# Create a non-root user
RUN adduser -D appuser
USER appuser

# Run the application
CMD ["./app"]
```

### 5. Use .dockerignore

Create a `.dockerignore` file to exclude unnecessary files from the build context:

```
.git
.github
*.md
*.log
test/
docs/
```

## Customizing Scotter-Generated Container Files

### Go Projects

The Go container template uses a multi-stage build approach:

```dockerfile
# Build stage
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY go.mod go.sum* ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /app/bin/myapp ./cmd/myapp

# Final stage
FROM alpine:latest
WORKDIR /app
COPY --from=builder /app/bin/myapp /app/myapp
ENTRYPOINT ["/app/myapp"]
```

Common customizations:

1. **Adding build arguments**:
   ```dockerfile
   ARG VERSION=dev
   RUN go build -ldflags="-X main.Version=${VERSION}" -o /app/bin/myapp ./cmd/myapp
   ```

2. **Including additional runtime dependencies**:
   ```dockerfile
   RUN apk add --no-cache ca-certificates tzdata
   ```

3. **Setting environment variables**:
   ```dockerfile
   ENV APP_ENV=production
   ```

### Shell Projects

For shell projects, consider these customizations:

1. **Installing additional packages**:
   ```dockerfile
   RUN apk add --no-cache bash curl jq
   ```

2. **Setting default environment variables**:
   ```dockerfile
   ENV LOG_LEVEL=info
   ```

3. **Adding health checks**:
   ```dockerfile
   HEALTHCHECK --interval=30s --timeout=3s \
     CMD ["/app/bin/healthcheck.sh"] || exit 1
   ```

## GitHub Actions Integration

Scotter generates a GitHub Actions workflow for building and publishing container images. To enhance this workflow:

1. **Add image scanning**:
   ```yaml
   - name: Scan image for vulnerabilities
     uses: aquasecurity/trivy-action@master
     with:
       image-ref: 'ghcr.io/username/repo:latest'
       format: 'table'
       exit-code: '1'
       severity: 'CRITICAL,HIGH'
   ```

2. **Add multi-platform builds**:
   ```yaml
   - name: Set up QEMU
     uses: docker/setup-qemu-action@v2
   
   - name: Set up Docker Buildx
     uses: docker/setup-buildx-action@v2
   
   - name: Build and push
     uses: docker/build-push-action@v4
     with:
       platforms: linux/amd64,linux/arm64
       push: true
       tags: ghcr.io/username/repo:latest
   ```

## Testing Container Images

### Local Testing

Test your container image locally before pushing:

```bash
# Build the image
docker build -t myapp:test .

# Run the container
docker run -p 8080:8080 myapp:test

# Run tests against the container
curl http://localhost:8080/health
```

### Integration Testing

Add integration tests to your CI pipeline:

```yaml
- name: Run integration tests against container
  run: |
    docker run -d -p 8080:8080 --name test-container ghcr.io/username/repo:${{ github.sha }}
    sleep 5
    ./scripts/integration-tests.sh
    docker stop test-container
```

## Common Issues and Solutions

### Build Context Issues

If your build context is too large:
1. Use `.dockerignore` to exclude unnecessary files
2. Build from a specific subdirectory if possible

### Permission Denied Errors

If you encounter permission issues:
1. Check file permissions in your repository
2. Ensure scripts are executable (`chmod +x`)
3. Use `COPY --chmod=0755` in your Dockerfile

### Container Startup Failures

If your container fails to start:
1. Add proper error handling in your entrypoint script
2. Include logging to help diagnose issues
3. Consider adding a health check
