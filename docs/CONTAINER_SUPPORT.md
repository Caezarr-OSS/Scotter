# Container Support in Scotter

Scotter provides robust support for containerization, allowing you to easily generate container configuration files for your projects. This document explains how container support works in Scotter and how to use it effectively.

## Overview

Containerization is a powerful approach to packaging applications and their dependencies into a standardized unit for development, shipment, and deployment. Scotter supports both Docker and Podman/OCI container formats, giving you the flexibility to choose the tooling that best fits your workflow.

## Container File Formats

Scotter supports two container file formats:

1. **Dockerfile** - The standard format used by Docker
2. **Containerfile** - The standard format used by Podman and other OCI-compliant container engines

### Technical Differences

While both formats use identical syntax and directives, they differ in name convention:

- Docker by default looks for a file named `Dockerfile`
- Podman and other OCI-compliant tools by default look for a file named `Containerfile`

The content of both files follows the same OCI (Open Container Initiative) specification, meaning you can use the same instructions and directives regardless of which format you choose.

## How It Works in Scotter

When you initialize a project with Scotter and select the container feature, the following happens:

1. You'll be prompted to choose between Dockerfile and Containerfile formats
2. Based on your project's programming language, Scotter selects the appropriate template
3. The template is processed with your project's configuration
4. The resulting container file is saved with the name you selected (either `Dockerfile` or `Containerfile`)
5. If GitHub Actions is enabled, a workflow for building and publishing container images is also generated

## Template Selection

Scotter selects templates based on your project's programming language:

### Go Projects

For Go projects, Scotter uses a multi-stage build approach:
- First stage builds the Go binary with proper optimization flags
- Second stage creates a minimal runtime image with just the binary
- This results in a small, secure container image

### Shell Projects

For shell-based projects:
- Uses Alpine Linux as a lightweight base
- Copies your shell scripts and makes them executable
- Sets up the entrypoint to your main script

### Default Template

For other project types:
- Uses Alpine Linux as a lightweight base
- Copies all project files
- Provides a basic shell entrypoint

## GitHub Actions Integration

When you enable both container and GitHub Actions features, Scotter generates a workflow file (`container.yml`) that:

1. Builds your container image on push to main/master branches
2. Tags the image with the Git SHA and 'latest'
3. Pushes the image to GitHub Container Registry (ghcr.io)

## Customizing Container Files

After Scotter generates the initial container file, you can customize it to fit your specific needs:

- Add additional dependencies
- Configure environment variables
- Set up volumes and ports
- Optimize build steps

## Best Practices

When working with the container files generated by Scotter:

1. **Keep images small**: Avoid installing unnecessary packages
2. **Use multi-stage builds**: Separate build and runtime environments
3. **Leverage caching**: Order your instructions to maximize cache usage
4. **Set proper permissions**: Avoid running containers as root when possible
5. **Pin base image versions**: Use specific tags rather than 'latest'

## Troubleshooting

Common issues when working with container files:

1. **Build context errors**: Make sure you're building from the correct directory
2. **Permission issues**: Check file permissions, especially for executable scripts
3. **Missing dependencies**: Ensure all required packages are installed in the container
